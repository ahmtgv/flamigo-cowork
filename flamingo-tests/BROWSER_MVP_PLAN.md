**Статус:** тест-набор spec-полный (92 теста, 13 файлов, типы чистые). Против живого stage ещё не гонялся — нужен запущенный фронтенд.
**Цель этого документа:** (1) что закрыто, (2) что должна дать команда, чтобы набор пошёл зелёным, (3) проактивные сценарии разработки, которые я рекомендую заложить в сам продукт.

## 2. Что должна дать команда, чтобы набор пошёл зелёным

Три вещи. Тесты написаны, но им не на что «опереться», пока их нет.

### 2.1. `data-testid` на элементах (главное)

Все селекторы вынесены в `tests/config/flamingo.ts → testIds`. Фронтенд проставляет их один раз. **Чеклист для фронтенда** (полный список — в конфиге):

```
Вход/аккаунт:  login-form, register-form, forgot-password-link,
               reset-password-form, twofactor-input,
               session-expired-notice, logout-all-button
Кабинет:       cabinet-dashboard, progress-widget, schedule-widget,
               materials-list, continue-learning-button
Админ:         admin-panel, manage-courses-button, manage-groups-button,
               students-table, reports-panel, export-data-button
SEduM:         sedum-session, attention-badge, fatigue-hint, sedum-trend,
               privacy-indicator, camera-denied-notice,
               pause-session-button, resume-session-button
Согласие:      consent-dialog, consent-accept-button, consent-decline-button
```

Это работа на несколько часов, которая делает тесты нехрупкими: они переживают рефакторинг вёрстки.

### 2.2. Тестовое окружение и сид данных

- Изолированный stage с сидируемой БД (не общий, который ломают руками).
- Скрипт `seed:test`: `student_1`, `student_2`, `teacher_1`, `admin_1`, `parent_1` — через API, не через UI.
- Заполнить `.env` (`BASE_URL`, `BACKEND_ORIGIN`, `TEST_PASSWORD`, `*_ID`).

### 2.3. Видео-фикстуры SEduM

- 5 коротких роликов в `tests/fixtures/source/` (см. `scripts/make-fixtures.sh`): смотрит в экран / отводит взгляд / нет лица / глаза закрыты / слабый свет.
- `npm run fixtures` соберёт Y4M (с патчем `C420`).

**Как только эти три пункта готовы** — `npm test`, и набор реально прогоняется. Порядок первого запуска: сначала `npm run test:privacy` и `npm run test:perms` (это блокеры), потом остальное.

## 3. Проактивные советы: что заложить в САМ продукт (не в тесты)

Тесты выше проверяют поведение, которого у вас, возможно, ещё нет. Это и есть требования к разработке браузерной версии — рекомендую внести их в бэклог.

### 3.1. SEduM — деградация, а не поломка
- **Отказ камеры не должен ломать урок.** Если ученик не дал доступ — урок работает, SEduM молча выключается, показывается спокойное `camera-denied-notice`. Тест `обработка отказа камеры` это ждёт. Это и UX, и юридически чистый путь (принуждать к камере нельзя).
- **Пауза = полная остановка отправки.** Кнопка паузы должна реально прекращать и анализ, и отправку агрегатов, и прятать индикатор приватности. Проверяется тестом.
- **Индикатор камеры виден всегда, пока идёт анализ.** Ученик/родитель обязаны видеть, что камера активна. Это требование доверия и прозрачности (UNESCO/OECD).

### 3.2. Тренд, а не ярлык (этическая рамка)
- Экран `sedum-trend` подаёт данные как **эволюционирующий тренд с языком роста**, без «диагнозов». Тест прямо падает на словах «диагноз», «неспособен», «СДВГ», «отсталость». Это защищает вас от главного репутационного и правового риска профилирования детей.
- Формулировки: «на этой неделе больше сосредоточенности на…», «тренд за 4 недели» — не «невнимательный ученик».

### 3.3. Приватность как контракт
- Определите **JSON-схему агрегата** SEduM и держите её как контракт (`allowedSedumEndpoints`, `MAX_AGGREGATE_BYTES`). Всё, что не проходит схему, privacy-gate считает нарушением.
- Агрегаты шлите **по WebSocket или маленькими POST** — но никогда бинарь/кадры. Тест ловит и то, и другое.

### 3.4. Аккаунт
- **Сброс пароля без энумерации:** одинаковый ответ для существующего и несуществующего адреса.
- **Истёкшая сессия → редирект на вход**, а не «белый экран» (частый баг SPA).
- **Троттлинг входа** после N неудач.
- **Согласие родителя в регистрации ученика** — обязательный шаг (152-ФЗ).

### 3.5. Роли
- Каждый эндпоинт — строка в `permissionMatrix`. Правило: **отказ не «протекает» чужим идентификатором** в теле ошибки.
- Отчёты в админке — только агрегаты; сырых кадров нет ни в UI, ни в API (`/api/sedum/raw/*` → 403/404).

## 4. Рекомендуемая последовательность (браузер → устройства)

```
1. Фронтенд проставляет data-testid          (несколько часов)
2. Seed тестовых пользователей + .env         (½ дня)
3. Снять 5 видео-фикстур + npm run fixtures    (½ дня)
4. Прогнать privacy-gate + permissions         ← первые зелёные блокеры
5. Дозакрыть продуктовые пробелы из раздела 3  (итеративно, с тестами как ТЗ)
6. Полный зелёный набор в CI на каждый PR
7. Зафиксировать браузер как ЭТАЛОН сценариев  → уходим на устройства
```

Пункт 7 важен: браузерный набор становится **живой спецификацией** того, что значит «правильный Flamingo». Нативные харнессы (iPad/Android/Windows/macOS) переиспользуют те же сценарии, фикстуры и privacy-контракт — не изобретая заново.

## 5. Definition of Done для браузерной версии

- [ ] Все `data-testid` проставлены, `npm test` идёт без «не найден элемент».
- [ ] `privacy-gate` и `permissions` — зелёные и блокируют merge.
- [ ] Все 5 SEduM-фикстур дают ожидаемое дискретное состояние.
- [ ] Отказ камеры и пауза обрабатываются по разделу 3.1.
- [ ] Экран тренда проходит проверку «не диагноз».
- [ ] Набор проходит 20 прогонов подряд (`npm run test:flake`) без флака.
- [ ] CI зелёный на каждый PR; privacy-отчёт сохраняется артефактом.

Когда чеклист закрыт — браузерная версия «добита», и переход на устройства идёт по `PLATFORM_TEST_MATRIX.md`, где браузер уже служит эталоном.

## 6. Риски и на что смотреть

- **Флак камеры.** Инициализация fake-камеры + первый кадр SEduM — самое медленное место. Первому ожиданию `attention-badge` дан таймаут 20 с; не уменьшать.
- **Y4M-фикстуры тяжёлые.** Только 480p, ≤15 с, Git LFS для исходников.
- **Тренд-тест — «страховка от overclaim».** Если продукт-команда добавит «умную» формулировку — тест напомнит про этическую рамку. Это фича, не помеха.
- **Не тестировать эмоции.** В браузерном наборе намеренно нет проверок «распознавания эмоций» — этого не должно быть в продукте (запрет EU AI Act в образовании + научная несостоятельность). Если такой функционал появится — это сигнал остановиться и пересмотреть, а не писать на него тест.

# Flamingo — платформенная матрица тестирования

**Платформы продукта:** планшеты (+ фолды), нативный десктоп (Windows/macOS), браузер.
**Реализация:** всё нативное (общего UI-кода между платформами нет).
**Дата:** 21 июля 2026

---

## 0. Честная рамка: что означает «всё нативное»

Вы выбрали нативную реализацию на каждой платформе. Это даёт лучший UX и доступ к камере/сенсорам, но для тестирования это **самый дорогой сценарий**: нет общего UI-кода → нельзя переиспользовать один тест-набор, каждый таргет требует своего харнесса.

**Что удерживает стоимость под контролем — три общих слоя, которые НЕ дублируются:**

| Общий слой | Где живёт | Что переиспользуется |
|---|---|---|
| **Сценарии** (что проверяем) | `tests/config/` как спецификация | Один список путей/ролей/прав — на все платформы |
| **`FrameSource` seam** | паттерн на каждом языке | Одинаковая архитектура подмены камеры везде |
| **Golden-фикстуры** (видео + ожидания) | `tests/fixtures/` | Одни и те же ролики и ожидаемые состояния SEduM |
| **Privacy-контракт** | JSON-схема агрегата | Один контракт «что можно слать», проверяется на всех таргетах |

Тесты пишутся на 5 харнессах, но **думаете вы один раз**. Это принципиально: без общих сценариев/фикстур пять команд начнут расходиться в том, что вообще считается «правильным поведением SEduM».

> Одно наблюдение по стоимости (не давлю, продуктовое решение ваше): если десктоп когда-нибудь станет Electron, а планшет — WebView/PWA поверх той же веб-кодовой базы, тест-стоимость падает в разы — Playwright покрыл бы и их, и проблема отсутствия камеры в iOS Simulator исчезла бы. Пока всё нативное — планируем по-дорогому, но аккуратно.

## 1. Инструмент на каждый таргет (проверено, 2026)

| Таргет | Движок тестов | Камера/SEduM | Статус инструмента |
|---|---|---|---|
| **Браузер** | **Playwright** (Chromium/Firefox; WebKit — ручной) | ✅ fake-camera флаги + Y4M | зрелый |
| **iPad (нативный)** | **XCUITest** (Swift) или Maestro | ⚠️ seam + real device (в Simulator камеры нет) | зрелый |
| **Android tablet (нативный)** | **Espresso** или **Maestro** | ⚠️ seam + video injection на реальном | зрелый |
| **Windows (нативный)** | **FlaUI** (.NET) или pywinauto | ⚠️ seam + OBS virtual cam | FlaUI акт. поддержка |
| **macOS (нативный)** | **XCUITest** (Swift) | ⚠️ seam + OBS virtual cam | зрелый |

**Красные флаги, которых избегаем:**
- ❌ **WinAppDriver** — мёртв (нет релиза с 2020, исходники сервера закрыты). Только FlaUI/pywinauto.
- ⚠️ **Appium Mac2** — работает, но «сырой» (лучший локатор XPath, не зрел). Для нативного macOS — XCUITest.
- ⚠️ **WebKit/Safari** — подмены камеры нет. Ручное или реальное устройство.

## 2. Стратегия камеры/SEduM по платформам

Ключевой факт остаётся: **у iOS Simulator камеры нет вообще.** Поэтому `FrameSource` (абстракция источника кадров, куда в тестовой сборке подставляется фикстурное видео) — **обязателен на iPad и macOS**, и применяется единообразно везде.

| Платформа | Подмена камеры в CI | Реальная камера |
|---|---|---|
| Браузер | fake-device + Y4M-фикстура (флаги Chromium) | — |
| iPad | `FileFrameSource` (Swift) — фикстура в сборке | BrowserStack **video injection**, ночью |
| Android tablet | `FileFrameSource` (Kotlin) | BrowserStack video injection, ночью |
| Windows | `FileFrameSource` (.NET) | OBS virtual cam, дымовой прогон |
| macOS | `FileFrameSource` (Swift) | OBS virtual cam, дымовой прогон |

**Правило ассертов везде одинаковое:** проверяем дискретное состояние («Отвлечён») и UI-последствие, никогда — float-координаты. Golden-фикстуры и ожидаемые состояния — общие (`tests/config/flamingo.ts → videoFixtures`).

> ⚠️ Спайк перед закупкой BrowserStack: убедитесь, что их video injection перехватывает непрерывный `AVCaptureVideoDataOutput` (их API-список фото-центричный), иначе iPad-камера не покроется на реальных устройствах. У Sauce Labs видео-инъекции нет.

## 3. Фолды — базовое покрытие (как вы просили)

Два уровня, оба лёгкие:

**A. Браузерная версия — уже в каркасе.** Проекты `tablet-landscape`, `foldable-folded` (673×841), `foldable-unfolded` (1812×2176) гоняют `responsive.spec.ts`:
- нет горизонтального переполнения на узком/сложенном экране;
- навигация доступна на любом размере;
- индикатор приватности SEduM виден и в сложенном виде;
- **continuity:** состояние урока сохраняется при resize «сложил → разложил».

**B. Нативный Android tablet — фолд через ADB.** BrowserStack и Espresso умеют менять постуру реального устройства:
```bash
# сложить / разложить в ходе теста
adb shell cmd device_state state folded
adb shell cmd device_state state unfolded
```
Базовый набор (не углубляясь):
- после разложения layout перестроился в широкий, без обрезаний;
- **сессия SEduM не прервалась** при смене постуры (камера не «отвалилась»);
- введённый текст / позиция в уроке сохранились (continuity).

iPad-фолдов нет (iPad не складывается), так что там — только адаптив портрет/ландшафт.

## 4. Что общее, что дублируется (карта усилий)

```
                    ┌───────────────────────────────────────┐
   ОБЩЕЕ (1 раз):   │ Сценарии · FrameSource-паттерн ·       │
                    │ Golden-фикстуры · Privacy-контракт     │
                    └───────────────────────────────────────┘
                                     │
        ┌────────────┬───────────────┼───────────────┬───────────────┐
        ▼            ▼               ▼               ▼               ▼
   Браузер       iPad          Android tab       Windows          macOS
   Playwright    XCUITest      Espresso/Maestro  FlaUI            XCUITest
   [готово]      [seam+спека]  [seam+спека]      [seam+спека]     [seam+спека]
```

- **Vision-пайплайн (golden-кадры)** тестируется **один раз**, отдельно от UI, на CPU. Модель одна на все платформы → и тест один. Не дублируется по таргетам.
- **Privacy-gate** реализуется на каждом таргете (перехват трафика), но **логика и контракт — общие**.
- **UI E2E** — единственное, что честно дублируется 5 раз. Здесь и лежит основная стоимость.

## 5. Приоритеты внедрения

| Очередь | Что | Почему |
|---|---|---|
| **1** | Браузер (Playwright) — **уже собран** | Дешевле всего, покрывает наибольшую долю логики, служит эталоном сценариев |
| **2** | Vision-пайплайн golden-тесты | Один раз на все платформы; ловит регрессии ядра SEduM |
| **3** | `FrameSource` в проде на всех нативных таргетах | Блокер тестируемости; сделать до нативных E2E |
| **4** | Android tablet (Maestro) + iPad (XCUITest) E2E основных путей | Планшет — основная поверхность SEduM |
| **5** | Windows (FlaUI) + macOS (XCUITest) E2E | Меньшая аудитория, но нативная камера — тот же seam |
| **6** | Фолды базово (ADB) + реальные устройства ночью | По вашему приоритету — базово |

## 6. Влияние на стоимость (грубо, поверх прошлой оценки)

Прошлая оценка (~$7–12k/год) покрывала **web + гипотетический общий мобильный слой**. Всё-нативное поднимает планку:

| Позиция | Комментарий |
|---|---|
| Инструменты | по-прежнему **$0** (Playwright/Maestro/Espresso/XCUITest/FlaUI — все бесплатны) |
| Реальные устройства (гриды) | +$199/параллель/мес за камеру на реальных iPad/Android; фолд-устройства — там же |
| macOS CI-раннеры | дороже Linux (~×10) — нужны для XCUITest iPad/macOS |
| **Инженерное время** | главный рост: **5 харнессов вместо 1–2**. Это не лицензии, это люди |

Вывод: денежная часть растёт умеренно (гриды + macOS-раннеры), а вот **человеко-часы** — существенно. Отсюда важность общих сценариев/фикстур из раздела 0: они не убирают 5 харнессов, но не дают им разъехаться и дублировать мышление.

# Ранний прототип на бесплатном хосте — план и алгоритм

**Идея (ваша):** до деплоя поднять кликабельный прототип на бесплатном хосте, без авторизации — вход по ссылкам-ролям (учитель / ученик / учебное заведение / родитель / админ). Так появляется на чём реально гонять тесты и что показывать.

**Оценка: одобряю, с одним жёстким ограничением (ниже).**

---

## ⚠️ 0. Ограничение, нарушать нельзя

Публичный прототип с ролями и SEduM (камера) **не должен содержать:**
- реальных данных учеников/детей (только синтетические, выдуманные);
- реального биометрического анализа с отправкой на сервер (SEduM — замокан или полностью on-device без отправки).

Причина: как только на бесплатный публичный хост попадут настоящие несовершеннолетние или реальная обработка камеры — это обработка ПДн/биометрии детей вне контура согласия и защиты → нарушение 152-ФЗ. Для кликабельного каркаса с фейковыми данными это не действует.

**Обязательные «предохранители» на прототипе:**
- баннер «DEMO — тестовый прототип, не для реальных данных»;
- `<meta name="robots" content="noindex, nofollow">` + `X-Robots-Tag` (чтобы не индексировался);
- базовая защита от случайных посетителей: Cloudflare Access / простой пароль на весь сайт (у Cloudflare Pages это бесплатно);
- камера в demo — либо выключена, либо анализ идёт локально и ничего не шлёт; никаких кадров на сервер.

## 1. Где размещать (проверено, 2026)

| Что | Рекомендация | Почему |
|---|---|---|
| **Хостинг** | **Cloudflare Pages** | Безлимитный трафик и запросы на free-тарифе, 500 сборок/мес, глобальный CDN, бесплатный HTTPS, встроенный Access (пароль на сайт). Для SPA/статики — лучший бесплатный вариант. |
| Альтернатива | Vercel (если фронт на Next.js) / Netlify | Отличный DX для React/Next; free 100 ГБ трафика. |
| **Домен (бесплатно)** | Поддомен хоста: `flamingo-demo.pages.dev` | Мгновенно, стабильно, HTTPS из коробки. **Freenom мёртв** — бесплатные `.tk/.ml` больше не работают, не связываться. |
| Домен (почти бесплатно) | ~$1–10/год на Namecheap (`flamingo.app` и т.п.) | Для показов инвесторам надёжнее «настоящего» имени, чем нестабильные free-домены. |
| Условно-бесплатно | `is-a.dev` / `js.org` (через PR) | Только если нужен красивый бесплатный поддомен; approval занимает время. |

**Рекомендация:** старт на `*.pages.dev`, а когда понадобится показывать наружу — купить дешёвый домен (это надёжнее любого «free»).

## 2. Пункт «2», который вы оставили пустым — предлагаю

Между «отгрузить на хост» и «убрать вход» логично встаёт:

> **2. Синтетические demo-данные + замоканный SEduM + авто-деплой из Git.**

То есть: наполнить прототип фейковыми курсами/учениками/оценками, SEduM показывает заранее заготовленный «тренд» без реальной камеры, а каждый пуш в `main` автоматически деплоится на Pages. Без этого «кликабельный прототип» будет пустым и недемонстрируемым.

## 3. Вход по ссылкам-ролям (без авторизации) — как делать

**Маршруты вместо логина:**

```
/r/student      → кабинет ученика
/r/teacher      → кабинет преподавателя
/r/school       → кабинет учебного заведения
/r/parent       → кабинет родителя
/r/admin        → админ-панель
/               → «выбор роли» (5 больших ссылок) + demo-баннер
```

**Механика:** маршрут `/r/<role>` кладёт `localStorage.demo_role = <role>` и грузит соответствующий кабинет с синтетическими данными. Настоящего токена и бэкенда нет.

**Переключатель ролей** в углу — чтобы на демо мгновенно прыгать между кабинетами (удобно инвесторам и QA).

**Важно — это только demo.** В реальной версии вход вернётся (регистрация, согласие родителя по 152-ФЗ, роли через бэкенд). Ссылки-роли — строго прототипный обходной путь, не архитектура продукта.

## 4. Тесты уже адаптированы под этот режим

Я добавил в харнесс **`demo`-режим** (не только план — код):
- `authMode: 'demo'` в `tests/config/flamingo.ts` — вместо логина `auth.setup` открывает `/r/<role>` и сохраняет состояние роли;
- команда **`npm run test:demo`** — гоняет только те проекты, что осмысленны без бэкенда: кабинеты ученика/преподавателя, SEduM (на моках), адаптив/фолды, доступность.

Что в demo **не** гоняем (нужен реальный бэкенд, вернём после появления stage): `privacy-gate`, `permissions/IDOR`, backend-части аккаунта (logout-all, троттлинг). Это честная граница: права доступа и утечку биометрии нельзя проверить без сервера.

## 5. Алгоритм дальнейших действий (пошагово)

### Фаза A — Каркас прототипа (2–3 дня)
1. Выбрать веб-стек прототипа (рекомендую то, на чём будет реальная веб-версия; если не решено — React + Vite, деплой на Pages в один клик).
2. 5 маршрутов-ролей + экран выбора роли + demo-баннер + `noindex`.
3. Синтетические данные (fixtures): 2–3 курса, 5–10 «учеников», расписание, оценки, заготовленный SEduM-тренд.
4. Проставить `data-testid` из `tests/config/flamingo.ts` **сразу** — тогда тесты цепляются с первого дня.

### Фаза B — Деплой (0.5 дня)
5. Репозиторий на GitHub → подключить к Cloudflare Pages → авто-деплой из `main`.
6. Включить Cloudflare Access (пароль на весь сайт) + проверить HTTPS и `noindex`.
7. Прогнать `npm run test:demo` против `*.pages.dev` (заполнить `BASE_URL`).

### Фаза C — Итерации UX на реальном URL (1–2 недели)
8. Довести кабинеты по ролям до «показываемого» состояния.
9. Реализовать продуктовые сценарии SEduM из `BROWSER_MVP_PLAN.md` (деградация при отказе камеры, пауза, тренд «не диагноз») — на моках.
10. Собирать обратную связь (инвесторы/преподаватели) прямо на demo-URL.

### Фаза D — Появляется бэкенд/stage → выход из demo
11. Вернуть настоящий вход (регистрация, согласие родителя, роли через API).
12. Переключить `authMode` с `'demo'` на `'token'`/`'cookie'`.
13. Включить обратно `privacy-gate` и `permissions` — это блокеры перед реальными данными.
14. Только теперь допускать реальных пользователей/камеру (в защищённом контуре, с согласиями).

### Фаза E — Уходим на устройства
15. Браузерная версия зафиксирована как эталон сценариев → нативные харнессы по `PLATFORM_TEST_MATRIX.md`.

## 6. Чеклист готовности прототипа

- [ ] 5 ролей-ссылок работают, экран выбора роли есть
- [ ] Demo-баннер + `noindex` + пароль на сайт (Cloudflare Access)
- [ ] Только синтетические данные; SEduM замокан, кадры никуда не шлются
- [ ] `data-testid` проставлены
- [ ] Авто-деплой из `main` на Cloudflare Pages
- [ ] `npm run test:demo` зелёный против живого URL
- [ ] Нигде нет реальных данных детей

## 7. Риски

| Риск | Митигация |
|---|---|
| На публичный demo зайдёт реальный ученик и включит камеру | Пароль на сайт + камера в demo не шлёт данные + баннер |
| Прототип проиндексируется, «утечёт» как продукт | `noindex` + `X-Robots-Tag` + Access |
| Ссылки-роли останутся в проде как «архитектура» | Явно помечено: demo-обходной путь; в проде вход возвращается |
| Бесплатный домен отвалится (Freenom-кейс) | Поддомен хоста или дешёвый купленный домен, не «free TLD» |
| Demo-данные примут за реальные метрики | Баннер + водяной знак «DEMO» на дашбордах |

## Что нужно от вас, чтобы я двинул дальше

Один ответ разблокирует следующий шаг: **веб-стек прототипа** — берём React + Vite (быстрый старт, один клик на Cloudflare Pages), или у вас уже выбран фреймворк для реальной веб-версии, и прототип делаем на нём?

Как скажете — соберу каркас прототипа (5 ролей-ссылок, demo-данные, баннер, `data-testid`, конфиг авто-деплоя на Cloudflare Pages), и он сразу поедет под `npm run test:demo`.

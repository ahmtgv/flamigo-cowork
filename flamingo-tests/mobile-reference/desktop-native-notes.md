# Нативный десктоп (Windows / macOS) — тестирование и камера

Десктопные версии Flamingo — **нативные** (не Electron). Значит, общего кода
с вебом нет, и нужен отдельный тест-слой на каждую ОС.

## Инструменты (проверено, 2026)

| ОС | Брать | Не брать | Почему |
|---|---|---|---|
| Windows | **FlaUI** (.NET, v5.0, akт. поддержка) или **pywinauto** (Python) | ❌ **WinAppDriver** | У WinAppDriver нет релиза с ноября 2020, исходники сервера не опубликованы, 1100+ открытых issue. Для новых проектов мёртв. |
| macOS | **XCUITest** (Swift, прямо в Xcode) | ⚠️ Appium Mac2 | Mac2-драйвер работает (на XCTest), но «сырой»: лучший локатор — XPath, не зрел для серьёзной автоматизации. XCUITest надёжнее для нативного macOS. |

Если когда-нибудь появится общий слой (Electron/Tauri) — стратегия схлопнётся
к одному Playwright-набору. Пока таргеты нативные, это два отдельных харнесса.

## Камера для SEduM на десктопе

Тот же принцип, что и на мобильных: **не делать ОС-камеру зависимостью теста.**
Абстракция `FrameSource` (см. FrameSource.swift / .kt) применяется и здесь.

Два уровня подмены:

1. **App-layer (рекомендуется):** в test/debug-сборке `FrameSource` читает
   фикстурное видео. Детерминированно, без внешних зависимостей, работает в CI.
   На macOS это тот же `FrameSource.swift`, что и для iPad.
2. **OS-layer (для ручных/дымовых прогонов):** виртуальная камера на хосте.
   - Windows: **OBS Studio → Start Virtual Camera** (источник — зацикленный фикстурный ролик).
   - macOS: OBS Virtual Camera аналогично; приложение видит её как обычную камеру.
   Минус: недетерминированно, состояние хоста, плохо ложится в CI. Только как подстраховка.

## Пирамида на десктопе

| Слой | Чем | Частота |
|---|---|---|
| Vision-пайплайн (golden-кадры) | юнит-тесты на CPU | каждый PR |
| Логика сигнал → UI | FlaUI / XCUITest + `ScriptedFrameSource` | каждый PR |
| E2E основных путей | FlaUI / XCUITest + `FileFrameSource` (фикстура) | каждый PR |
| Реальная камера + разрешения ОС | ручной / дымовой на реальной машине | пред-релиз |

Privacy-gate на десктопе реализуется перехватом сетевого трафика приложения
(mitmproxy / прокси в test-сборке) с той же логикой, что и веб-версия:
в исходящих payload'ах не должно быть сырых биометрических признаков.
